0. What Is a Planning Problem?

A planning problem has four parts:

  1. Initial state — how the world looks right now
  2. Goal — how we want the world to look
  3. Actions — things we can do to change the world
  4. Constraints — rules about when actions can be used
Think of it like a puzzle. You have a starting configuration, a target configuration, and a set of allowed moves. Planning = finding the sequence of moves that gets you from start to target.

Preconditions and Effects

Each action has preconditions (what must be true before you can do it) and effects (what changes after you do it). Effects come in two types:

  • Add effects — things that become true (e.g., holding(X) after pickup)
  • Delete effects — things that stop being true (e.g., handempty after pickup)

Delete effects are crucial — they're what make planning hard, and they're what create the Sussman Anomaly.

1. Planning as Search

To find a plan, we search through possible states. Starting from the initial state, we try every possible action, see what new state we reach, then try every action from there, and so on — until we find a state that satisfies the goal.

Breadth-First Search (BFS) — The Simplest Approach

Explore states level by level: first all states 1 action away, then all states 2 actions away, etc. This guarantees finding the shortest plan, but it's extremely slow because it explores every possibility.

For Blocksworld with 4 blocks: ~4 possible actions per state, plan length ~8 → up to 48 = 65,536 states to explore. With 12 blocks it's astronomical.

BFS is like exploring a maze by trying every corridor equally. You'll definitely find the exit, but you waste time going down dead ends. What if you had a compass pointing roughly toward the exit? You'd get there much faster. That compass is called a heuristic.

What Is a Heuristic?

Heuristic = an educated guess about how far you are from the goal

Given any state, a heuristic function h(s) returns a number estimating "how many more actions do I probably need to reach the goal from here?"

A good heuristic makes search dramatically faster by telling us: "don't bother exploring that direction — it's far from the goal. Focus on this direction instead."

Example heuristics for Blocksworld:

  • Bad heuristic (h=0): "I have no idea how far the goal is." This gives no guidance — search degenerates to BFS.
  • Simple heuristic: "Count how many goal facts are NOT yet true." If the goal is on(A,B) AND on(B,D) and neither is true, h=2. This gives some guidance but is very rough.
  • hFF (what we use): "Solve a simplified version of the problem and use that solution's length." Much more accurate — explained below.

2. A* Search — The Algorithm We Used

A* is the gold standard for heuristic search. It combines two pieces of information:

f(s) = g(s) + h(s)

  • g(s) = the actual number of actions we've taken to reach state s (known exactly)
  • h(s) = the heuristic estimate of actions remaining to reach the goal (estimated)
  • f(s) = estimated total plan length if we go through state s

A* always expands the state with the lowest f(s) first. This means it prioritizes states that look like they're on a short path to the goal.

A* is like a GPS navigator. g(s) is the distance you've already driven. h(s) is the estimated remaining distance (straight-line to destination). f(s) is the estimated total trip time. The GPS always suggests the route with the lowest total, not just the one that looks closest right now.

What Does "Admissible" Mean?

Admissible = never overestimates

A heuristic is called admissible if it never guesses higher than the true cost. In other words, it's always optimistic.

Example: straight-line distance on a map is admissible because you can never drive somewhere in less distance than the straight line. The real driving distance is always ≥ the straight line.

Why it matters: If h(s) is admissible, then A* is guaranteed to find the optimal (shortest) plan. If h overestimates, A* might skip the optimal path because it looks too expensive.

Why "admissible"? The name

The term comes from the idea that such a heuristic is "admissible as evidence" — it's a valid, trustworthy estimate that won't mislead the search. An inadmissible heuristic might cause A* to "admit" (accept) a suboptimal solution because it misjudged the cost of the optimal one.

Think of it as a courtroom analogy: admissible evidence is reliable and can be trusted. Admissible heuristics are reliable and won't trick the search into making bad decisions.

3. The FF Heuristic (hFF) — How We Estimate Cost

What Does FF Stand For?

FF = Fast Forward. It's the name of a famous planner created by Jörg Hoffmann in 2001. The "FF heuristic" (hFF) is the heuristic function used inside that planner. We use it inside A* search.

The Core Idea: Delete Relaxation

Delete relaxation = pretend that actions never undo anything

Normally, when you pick up block A, you lose handempty (your arm is now full) and clear(A) and ontable(A). These are delete effects.

In the relaxed problem, we remove all delete effects. So after pickup(A), you have holding(A) BUT you also still have handempty, clear(A), and ontable(A). Facts only accumulate — nothing ever becomes false.

Imagine you have infinite robot arms instead of one. You can pick up A, but your arm is still "empty" for picking up B too. You can stack things without un-stacking anything first. It's a fantasy world where there are no trade-offs.

Why Is This Useful?

The relaxed problem is much easier to solve (polynomial time instead of exponential). And the length of the relaxed solution gives a reasonable estimate of the real cost — it's usually somewhat optimistic because the real problem has constraints the relaxed one ignores.

Worked Example

Current state: Initial state of instance-8. Goal: on(A,B) AND on(B,D).

Arm full, B not on C anymore, C is clear.		 unstack(B,C) — arm grabs B.
But arm is STILL empty too! B is STILL on C too! We just ADDED holding(B) and clear(C).
2 Must putdown(B) or stack(B,D) because arm is full. Can do ANYTHING — arm is "empty" (still true) so can pick up A, or stack B on D, or whatever.

In the relaxed world, we can achieve on(B,D) and on(A,B) much faster because we never lose handempty. The relaxed plan might be: unstack(B,C), stack(B,D), pickup(A), stack(A,B) — 4 steps. So hFF = 4 for the initial state.

Is hFF admissible? Not always! Sometimes the relaxed plan uses redundant actions (because it doesn't track deletes properly), making the estimate higher than the true cost. So hFF is not guaranteed admissible. This means A* with hFF might not find the truly optimal plan — but in practice it finds very good plans very fast. For our analysis, "near-optimal" is perfectly fine.

This is the Sussman Anomaly. The two goals on(A,B) and on(B,D) interact destructively: achieving the first one makes the second one harder. The key to solving it efficiently is recognizing the right order (bottom-up: B on D first, then A on B).

This asymmetry (harmful in one direction, harmless in the other) is exactly what defines the Sussman Anomaly. It means there's a required ordering: always do on(B,D) before on(A,B).

7. Landmark Orderings & Transitive Dependencies

Direct pairwise analysis (section 4-5) checks: "does achieving Gi alone make Gj alone harder?" But this misses transitive dependencies — goals that share no blocks can still require strict ordering through intermediate goals.

What is a Landmark?

A landmark is a fact that must become true at some point in every valid plan. For example, if your goal is on(A,B), then holding(A) is a landmark — there's no way to get A on B without first holding A.

Goal facts are always landmarks (they must be true at the end). But there are also intermediate landmarks — facts that must be achieved along the way.

What is a Necessary Ordering?

A necessary ordering L1 → L2 means: L1 must be achieved before L2 in every valid plan. This is provably correct — it's not a heuristic guess, it's a mathematical fact about the planning domain.

These orderings come from the formal method by Hoffmann, Porteous & Sebastia (2004).

Building a house: you must pour the foundation before framing the walls, and frame the walls before installing the roof. "Foundation → walls" and "walls → roof" are necessary orderings. Even though the foundation and roof don't directly interact (different materials, different workers), there's a transitive ordering: foundation → roof.
