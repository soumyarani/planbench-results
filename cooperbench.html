CooperBench — 5-Category Subgoal Interaction Analysis

652 task pairs from 12 open-source repositories (Python, TypeScript, Go, Rust). Two AI agents independently implement different features on a shared codebase. We analyze whether their patches conflict and map to the PlanBench planning taxonomy.

What are the subgoals?

Each CooperBench task assigns 2 agents to implement different features. Each agent produces a feature.patch — a unified diff against the base repository. Each agent’s patch = one subgoal. We analyze how these subgoals interact.

Methodology

Step 1: Extract Per-Agent Effects

Parse each agent’s feature.patch using PatchSymbolExtractor (regex-based extraction from diff hunks):

  • Write set: Symbols the agent ADDS/MODIFIES — function defs, class defs, method defs, imports, variable assignments, body modifications
  • Read set: Symbols the agent REFERENCES — function calls, name references, imports

All symbols qualified by file path (file::symbol) to avoid false positives across files.

Step 2: Compute Pairwise Interactions

SetDefinitionPlanBench Analog
write_writeSymbols BOTH agents writeShared resource (like shared block)
write_read_a→bSymbols A writes that B readsA’s effects ∩ B’s preconditions
write_read_b→aSymbols B writes that A readsB’s effects ∩ A’s preconditions

Step 3: Simulate Patch Ordering

For pairs with symbol conflicts, we simulate both orderings by checking hunk overlap: do both patches modify the same line ranges? If so, the second patch cannot apply after the first (context mismatch).

  • Both orderings clean → conflict is at symbol level only, may be resolvable
  • One ordering clean → DIRECT_DESTRUCTIVE (serializable)
  • Neither ordering clean → NON_SERIALIZABLE (needs semantic merge)

Step 4: Filter Idempotent Operations

Shared imports (typing.Optional, from X import Y) and __init__.py re-exports are idempotent — adding the same import twice has no effect. These are classified as SYNERGY (mutual benefit), not conflicts.

Concrete Examples

INDEPENDENT — outlines/task1371: feature1 vs feature2

Feature A: Add Custom Filters Support to Prompt Decorator
Feature B: Add Global Filter Registry for Template Reusability

Both modify outlines/prompts.py but write completely different symbols:

Agent A writes: _template_from_file, _template_from_str, create_jinja_env, from_file, from_str Agent B writes: register_filter, unregister_filter Write-Write: NONE A→B reads: NONE B→A reads: NONE

PlanBench analog: on(A,B) + on(C,D) with no shared blocks. delta(A→B) = 0, delta(B→A) = 0.

Note: CooperBench labels this CONFLICT because git sees overlapping lines, but at the symbol level there is no semantic interference.

SYNERGY — dirty_equals/task43: feature1 vs feature2

Feature A: Add IsIP validator
Feature B: Add IsMac validator

Both add new validator classes and both add the same re-exports to __init__.py:

Agent A adds to __init__.py: from ._other import FunctionCheck, IsIP, IsJson, IsUUID Agent B adds to __init__.py: from ._other import FunctionCheck, IsJson, IsMac, IsUUID Shared imports: FunctionCheck, IsJson, IsUUID (both agents re-export the same existing symbols)

PlanBench analog: Mutual synergy. Both agents assert the same imports — idempotent. Achieving A makes B’s import free (already done). delta(A→B) < 0, delta(B→A) < 0.

NON_SERIALIZABLE — outlines/task1371: feature1 vs feature3

Feature A: Add Custom Filters Support — rewrites prompt() to accept filters parameter
Feature B: Add Conditional Filter Loading — rewrites prompt() to accept conditional_filters parameter

Agent A rewrites: def prompt(fn=None, filters={}) # A's signature Agent B rewrites: def prompt(fn=None, conditional_filters=None) # B's signature Write-Write: [prompts.py::prompt, prompts.py::_template_from_file, ...] A writes, B reads: [prompts.py::prompt] B writes, A reads: [prompts.py::prompt] Simulation: A→B: A modifies lines 40-85, B also modifies lines 32-85 → CONFLICT (context mismatch) B→A: B modifies lines 32-85, A also modifies lines 40-85 → CONFLICT (context mismatch) NEITHER ordering works.

PlanBench analog: Sussman Anomaly. Both agents rewrite prompt() with incompatible signatures. Like on(A,B) + on(B,A) — mutual destruction. Requires semantic merge: combine A’s filters parameter with B’s conditional_filters into one function signature.

NON_SERIALIZABLE — dspy/task8394: feature1 vs feature2

Feature A: Add cache.bypass() context manager — modifies Cache.get() to check bypass flag
Feature B: Add cache namespaces — modifies Cache.get() to prefix namespace

Agent A adds to Cache.get(): if _cache_bypass_context.get(): return None # skip cache read Agent B adds to Cache.get(): namespace = self._namespace_var.get() if namespace: key = f"{ns_hash}:{key}" # prefix key with namespace Both modify Cache.get() body (lines 97-104) with different logic. Neither ordering applies — both patches expect the ORIGINAL get() body as context.

PlanBench analog: Both agents modify the same function body for different purposes. Like two goals both needing clear(B) and both consuming it. The merged solution must include BOTH bypass checking AND namespace prefixing in get().

Key Findings

  1. 69.3% of pairs are INDEPENDENT at symbol level even though CooperBench labels 76.5% as CONFLICT. Most “conflicts” are textual (overlapping lines) not semantic (different symbols).
  2. 24.1% are genuinely NON_SERIALIZABLE — both patches modify the same line ranges, so neither ordering of git apply works. These require semantic merge (understanding what both agents intended and combining them).
  3. 6.1% are SYNERGY — shared imports and registry re-exports that are idempotent (adding the same import twice is harmless).
  4. Only 0.5% are DIRECT_DESTRUCTIVE (one ordering works) — in code, when both agents touch the same lines, it’s almost always the case that NEITHER ordering works, unlike Blocksworld where you can always undo and redo.
  5. Code patches are “more non-serializable” than Blocksworld because patches have fixed line contexts. In Blocksworld, you can always unstack and rebuild (0% NS). In code, if both patches touch lines 40-85, neither can apply after the other (24.1% NS).

Why is CooperBench more non-serializable than Blocksworld?

In Blocksworld, achieving on(A,B) makes on(B,C) costlier but never impossible — you can always unstack A from B and try again. In code, once Agent A’s patch changes lines 40-85, Agent B’s patch (which expects the original lines 40-85 as context) simply cannot apply. There is no “undo” in git apply. This makes code patches inherently more prone to non-serializable interactions than classical planning domains.

Methodology Details

Tools

  • PatchSymbolExtractor: Regex-based extraction of function defs, class defs, imports, method defs from unified diff hunks
  • Hunk overlap simulation: Parse @@ -start,count +start,count @@ headers from both patches; check if line ranges overlap in the same file
  • Import filtering: Symbols matching typing.*, contextlib.*, re, json, etc. treated as idempotent (SYNERGY not DESTRUCTIVE)
  • Registry filtering: __init__.py re-exports treated as commutative list additions (SYNERGY)

Limitations

  • Regex-based extraction (not AST-level) — may miss some relationships
  • Python-only symbol extraction — Go/Rust/TypeScript get coarser analysis
  • No transitive analysis — cross-file import chains not traced (TRANSITIVE_DESTRUCTIVE = 0%)
  • Hunk overlap is a proxy for git apply failure — actual application may succeed in edge cases
