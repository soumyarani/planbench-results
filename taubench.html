τ-bench — 5-Category Subgoal Interaction Analysis

165 tasks (115 retail + 50 airline). An agent interacts with a simulated user to execute customer service operations. We analyze whether the goals (state-changing mutations) within each task interact.

What are the subgoals?

Each τ-bench task has a gold solution: a sequence of tool/API calls. Most calls are lookups (gathering information). The actual goals are the mutations — actions that change entity state (cancel order, modify address, return items). We analyze only goal-to-goal interactions, not lookup-to-lookup.

Key insight: Information gathering (find_user_id, get_order_details) is like handempty in Blocksworld — a reusable resource that is never consumed. Only entity state changes (order_pending → order_cancelled) create real interactions.

Methodology

Step 1: Identify Goals (Mutations) per Task

From each task’s gold solution, extract only state-changing actions:

Step 2: Model Entity State Effects

For each goal action, define what entity state it NEEDS, DELETES, and ADDS — exactly like PlanBench’s STRIPS model:

modify_pending_order_*(X)order_pending(X)—*_modified(X)

Only actions that DELETE state create destructive interactions. Modify/update actions NEED entity state but don’t consume it.

Concrete Examples

INDEPENDENT — Retail Task #4

Instruction: Modify items on two different orders

Goal 1: modify_pending_order_items(#W6247578) NEEDS: order_pending(#W6247578) DELETES: nothing Goal 2: modify_pending_order_items(#W4776164) NEEDS: order_pending(#W4776164) DELETES: nothing Different entities → no shared state → INDEPENDENT

PlanBench analog: on(A,B) + on(C,D) — different blocks, no interaction. Modifying order #W6247578 has zero effect on order #W4776164.

SYNERGY — Retail Task #41

Instruction: Change address and modify items on the same order #W4082615

Goal 1: modify_pending_order_address(#W4082615) NEEDS: order_pending(#W4082615) DELETES: nothing Goal 2: modify_pending_order_items(#W4082615) NEEDS: order_pending(#W4082615) DELETES: nothing Same entity, both need order_pending, NEITHER deletes it → SYNERGY

PlanBench analog: Two goals both needing clear(B) where neither action consumes it. Both goals can succeed in any order — shared precondition is preserved.

DIRECT_DESTRUCTIVE — Airline Task #34

Instruction: “You want to cancel your upcoming flights within reservation IDs XEHM4B and 59XX6W.”

The gold solution first updates flights on XEHM4B (downgrade from business), then cancels it:

Goal 1: update_reservation_flights(XEHM4B) NEEDS: reservation_active(XEHM4B) DELETES: nothing Goal 2: cancel_reservation(XEHM4B) NEEDS: reservation_active(XEHM4B) DELETES: reservation_active(XEHM4B) ← Goal 3: cancel_reservation(59XX6W) NEEDS: reservation_active(59XX6W) DELETES: reservation_active(59XX6W) Pair: Goal 1 vs Goal 2 → DIRECT_DESTRUCTIVE cancel DELETES reservation_active(XEHM4B) which update NEEDS. Must do update BEFORE cancel. Wrong order → update fails. Pair: Goal 1 vs Goal 3 → INDEPENDENT (different reservations) Pair: Goal 2 vs Goal 3 → INDEPENDENT (different reservations)

PlanBench analog: on(B,C) + on(A,B) chain — stack(A,B) deletes clear(B) which stack(B,C) needs. Must achieve on(B,C) before on(A,B). Same here: must update before cancel.

NON_SERIALIZABLE — Retail Task #13

Instruction: “You want to return all things in order #W5490111”

The gold solution returns items from the SAME order in two separate return calls:

Goal 1: return_delivered_order_items(#W5490111) [items A] NEEDS: order_delivered(#W5490111) DELETES: order_delivered(#W5490111) ← Goal 2: return_delivered_order_items(#W5490111) [items B] NEEDS: order_delivered(#W5490111) DELETES: order_delivered(#W5490111) ← MUTUAL DESTRUCTION: Both need order_delivered(#W5490111), both delete it. Goal 1 first → deletes order_delivered → Goal 2 cannot execute. Goal 2 first → deletes order_delivered → Goal 1 cannot execute. Neither ordering works.

PlanBench analog: on(A,B) + on(B,A) — mutual destruction. Both goals consume the same state. Requires combining both returns into a single API call (semantic merge).

Key Findings

  1. τ-bench is 82% INDEPENDENT at the goal level. Most multi-goal tasks operate on different entities (different orders, different reservations). The goals don’t interact at all.
  2. 16% SYNERGY — same entity, shared precondition, not consumed. Both goals can succeed in any order. The agent just needs to complete the shared lookups once.
  3. Only 2 pairs (1.4%) have destructive interactions — one DIRECT_DESTRUCTIVE (cancel after update) and one NON_SERIALIZABLE (double return on same order). τ-bench barely tests subgoal conflict reasoning.
  4. Previous analysis overcounted interactions by treating every ACTION pair (including lookups) as a subgoal pair. Lookups share information but information is never consumed — it’s like handempty in Blocksworld.

Why is τ-bench mostly independent?

τ-bench tasks are designed as customer service scenarios where a user has multiple unrelated requests (“cancel order A and return order B”). The goals naturally operate on different entities. PlanBench Blocksworld, by contrast, forces all goals to share the same small set of blocks — creating much richer interactions.

Analysis: PDDL-modeled entity state effects. Only mutations (state-changing actions) treated as goals. Information gathering filtered as reusable (non-consumable) resource. Taxonomy: Korf 1987 / Barrett & Weld 1994.
