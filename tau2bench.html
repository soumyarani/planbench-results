τ2-bench — 5-Category Subgoal Interaction Analysis

2,285 telecom troubleshooting tasks (Sierra Research, 2025). Unlike τ-bench where the agent executes API calls, in τ2-bench the user performs physical device actions (toggle settings, reboot, reseat SIM) guided by the agent. This is a Dec-POMDP where both agent and user modify shared state.

Methodology

What Are the Subgoals?

Each task has a gold solution: a sequence of user-side device actions that the agent should guide the user to perform. These are physical actions on the user’s phone:

Concrete Examples

DIRECT_DESTRUCTIVE — Airplane Mode Blocks Roaming

Task: [mobile_data_issue]airplane_mode_on|user_abroad_roaming_enabled_off

Scenario: User is abroad, mobile data not working. Airplane mode is stuck ON and roaming is disabled.

Gold solution (correct ordering): Step 1: [USER] toggle_airplane_mode() ← turn OFF airplane mode FIRST Step 2: [USER] toggle_roaming() ← NOW roaming can be enabled Wrong ordering: Step 1: [USER] toggle_roaming() ← FAILS: airplane mode blocks all connectivity Step 2: [USER] toggle_airplane_mode() ← too late, roaming toggle had no effect

PlanBench analog: on(B,C) + on(A,B) — must achieve on(B,C) first because stack(A,B) deletes clear(B). Here, airplane mode “deletes” connectivity, so it must be toggled OFF before any connectivity setting can work.

delta(airplane→roaming) > 0: Toggling airplane mode ON makes roaming toggle impossible (cost = ∞). Agent must guide user to turn airplane mode OFF first.

DIRECT_DESTRUCTIVE — Reboot Resets Toggle State

Task: [mobile_data_issue]data_mode_off|network_mode_wrong|apn_wrong

Scenario: User’s data isn’t working. Multiple issues: data toggled off, wrong network mode, wrong APN settings.

Gold solution (correct ordering): Step 1: [USER] reboot_device() ← reboot FIRST to clear cached state Step 2: [USER] reset_apn_settings() ← reset APN to defaults Step 3: [USER] set_network_mode_preference(auto) ← set correct network mode Step 4: [USER] toggle_data() ← enable data LAST Wrong ordering: Step 1: [USER] toggle_data() ← turn data on Step 2: [USER] set_network_mode_preference(auto) ← set network mode Step 3: [USER] reboot_device() ← DESTROYS steps 1 & 2! Step 4: [USER] reset_apn_settings() ← may work but data is off again

PlanBench analog: reboot_device is like a planning operator that DELETES multiple propositions at once (clears all toggles). Any toggle done BEFORE reboot gets undone. The agent must know: do destructive/resetting actions FIRST, then configure.

DIRECT_DESTRUCTIVE — SIM Reseat Drops Connections

Task: [mobile_data_issue]sim_card_loose|data_mode_off

Scenario: User’s SIM card is loose and data is disabled.

Gold solution: Step 1: [USER] reseat_sim_card() ← physically remove and reinsert SIM Step 2: [USER] toggle_data() ← enable data after SIM is properly seated If reversed: Step 1: [USER] toggle_data() ← may turn ON but SIM is loose = unreliable Step 2: [USER] reseat_sim_card() ← drops ALL connections, data toggle may be lost

PlanBench analog: reseat_sim_card is like putdown in Blocksworld — it temporarily destroys the current state to create a clean foundation. Must do physical resets BEFORE logical configuration.

INDEPENDENT — Different Toggles

Task: [wifi_calling_issue]data_mode_off|wifi_calling_disabled

Scenario: User needs both data and WiFi calling enabled.

Gold solution (either ordering works): Step 1: [USER] toggle_data() ← enable mobile data Step 2: [USER] toggle_wifi_calling() ← enable WiFi calling OR: Step 1: [USER] toggle_wifi_calling() ← enable WiFi calling Step 2: [USER] toggle_data() ← enable mobile data Both orderings work. Neither toggle affects the other.

PlanBench analog: on(A,B) + on(C,D) — different “blocks” (settings), no shared state. delta(A→B) = 0, delta(B→A) = 0.

SYNERGY — Same Permission Granted Twice

Task: [mms_issue]break_app_both_permissions

Scenario: Messaging app needs both storage and network permissions.

Gold solution: Step 1: [USER] grant_app_permission(storage) Step 2: [USER] grant_app_permission(network) Both are idempotent. Granting one doesn't affect the other. Both contribute to the goal (app working). Mutual synergy.

PlanBench analog: Like shared imports in CooperBench — both actions assert the same type of precondition without consuming any resource.

Key Findings

  1. τ2-bench has 22× more destructive interactions than τ-bench (15.8% vs 0.7%). Device physics creates real ordering constraints: airplane mode blocks toggles, reboot resets state, SIM reseat drops connections.
  2. The agent’s planning challenge is ORDERING: do destructive/resetting actions (reboot, SIM reseat, airplane mode OFF) BEFORE constructive actions (enable data, set network mode). Wrong ordering wastes user effort.
  3. 0% NON_SERIALIZABLE — there’s always a correct ordering for device actions. Unlike code patches (CooperBench) where neither ordering may work, device settings can always be configured in the right sequence.
  4. The Dec-POMDP aspect matters: the agent tells the user what to do, but the user executes. If the agent gives instructions in the wrong order, the user’s actions undo each other — and the agent may not know until the user reports failure.

Why τ2-bench has more destructive interactions

τ-bench goals operate on different entities (different orders/reservations) — cancelling order A doesn’t affect order B. In τ2-bench, all actions operate on the same device — rebooting affects ALL settings. This is like Blocksworld where all goals share the same small set of blocks, creating dense interactions.
