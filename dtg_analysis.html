Domain Transition Graphs (DTG) — Structural Analysis

A Domain Transition Graph reveals the structure hidden inside each benchmark domain. By mapping how state variables (fluents) transition between values, we can systematically identify where hard interactions live — and where benchmarks fail to test them.

Section 1: What is a DTG?

Definition

A Domain Transition Graph (DTG) for a state variable (fluent) is a directed graph where each node is a possible value of that variable, and each edge is an action that causes the transition from one value to another. DTGs were introduced by Helmert (2006) for the SAS+ planning formalism and are the backbone of causal graph heuristics in modern planners.

In τ-bench, fluents are things like order_status ∈ {pending, shipped, delivered, cancelled, returned, exchanged}. Each API action (cancel_order, return_items, exchange_items) is an edge in this graph.

Example: order_status DTG

order_status DTG ================ pending ──[modify_*]──→ pending (self-loop: non-destructive) │ ├──[cancel]────────→ cancelled (ABSORBING — no outgoing edges) │ └──[ship]──────────→ shipped │ └──[deliver]──→ delivered │ ├──[return]───→ returned (ABSORBING) │ └──[exchange]─→ exchanged (ABSORBING) Legend: ──→ = action causing state transition (ABSORBING) = terminal state, no way to leave

Key DTG Concepts

ConceptDefinitionExample
Absorbing stateOnce entered, cannot leave. No outgoing edges exist.cancelled, returned — you cannot un-cancel an order
Self-loopAction that transitions a variable to its current value (no net change).modify_items on a pending order: status stays pending
ForkTwo edges from the same node to different absorbing states. Can only take ONE.return OR exchange from delivered — mutually exclusive
Conditional transitionEdge exists only when another fluent has a specific value.toggle_data only works when airplane_mode = off
Cross-DTG side effectAction on one fluent causes a transition on a different fluent.reboot_device resets toggle states (cross-fluent cascade)
Counter fluentNumeric variable that only decreases (or only increases). Monotonic.Gift card balance (decreases), PIN attempts remaining (decreases)

Why DTGs matter for benchmark design

DTGs expose the topology of a domain. A domain with only self-loops and no absorbing states (like modify-only operations) can never produce destructive interactions. A domain with forks to absorbing states creates mutual exclusion. A domain with cross-DTG side effects creates ordering chains. By analyzing DTGs, we can see exactly what interaction types a benchmark can produce — and what it structurally cannot.

Why telecom is structurally richer

Unlike retail and airline where goals operate on different entities (different orders, different reservations), telecom actions ALL operate on the SAME device. Every toggle, every setting change, every reboot affects the same shared state. This means virtually every pair of goals shares state — creating dense cross-DTG dependencies.

Section 3: 12 Hard Instance Designs

Each instance targets a specific gap identified by DTG analysis. These are designed to be structurally hard — the difficulty comes from the topology of the state space, not from ambiguous language or obscure domain knowledge.

DIRECT_DESTRUCTIVE hard_01: Precondition Clobber MEDIUM

Description: Cancel and modify the same pending order. Cancel destroys the precondition that modify needs.

User instruction: "I need to cancel order #W1234 and also change its shipping address to 123 Main St."

Goal A: cancel_order(#W1234) PRE: order_pending(#W1234) DEL: order_pending(#W1234) ADD: order_cancelled(#W1234) Goal B: modify_order_address(#W1234, "123 Main St") PRE: order_pending(#W1234) DEL: — ADD: address_updated(#W1234)

DTG analysis: Both goals originate from the pending node. Goal A takes the edge to cancelled (absorbing). Goal B is a self-loop at pending. If A executes first, the state moves to cancelled and B's precondition order_pending is gone. The only valid ordering is B before A.

Correct ordering: B then A (modify first, then cancel). The agent must recognize that cancellation is irreversible.

NON_SERIALIZABLE hard_02: Mutual Destruction HARD

Description: Return and exchange the same delivered order. Both consume order_delivered and transition to different absorbing states.

User instruction: "I want to return items A and B from order #W5678, and exchange item C from the same order."

Goal A: return_items(#W5678, [A, B]) PRE: order_delivered(#W5678) DEL: order_delivered(#W5678) ADD: order_returned(#W5678) Goal B: exchange_items(#W5678, [C]) PRE: order_delivered(#W5678) DEL: order_delivered(#W5678) ADD: order_exchanged(#W5678)

DTG analysis: This is the fork from delivered. Both edges lead to absorbing states (returned, exchanged). Both delete order_delivered. A before B: B fails. B before A: A fails. No valid serialization exists.

Correct resolution: Semantic merge — combine into a single API call that processes all items (returns A,B and exchanges C simultaneously). The agent must detect the mutual exclusion and propose the merge.

DIRECT_DESTRUCTIVE hard_03: Resource Consumption HARD

Description: Two orders share the same gift card with $100 balance. Order A costs $70 and Order B costs $50. Total exceeds balance.

User instruction: "Apply gift card GC-9999 (balance: $100) to order #W1111 ($70 total) and order #W2222 ($50 total)."

Goal A: apply_gift_card(GC-9999, #W1111, $70) PRE: gift_card_balance(GC-9999) >= 70 DEL: gift_card_balance(GC-9999, $100) ADD: gift_card_balance(GC-9999, $30) Goal B: apply_gift_card(GC-9999, #W2222, $50) PRE: gift_card_balance(GC-9999) >= 50 DEL: gift_card_balance(GC-9999, $100) ADD: gift_card_balance(GC-9999, $50) Counter fluent: gift_card_balance ∈ {$0, $1, ..., $100} (monotonically decreasing)

DTG analysis: The gift card balance is a counter fluent — it only decreases. A before B: balance becomes $30, B needs $50 → B fails. B before A: balance becomes $50, A needs $70 → A fails. Both orderings fail because combined demand ($120) exceeds supply ($100).

Correct resolution: Agent must detect insufficient shared resource and ask user to prioritize which order gets the gift card, or split the payment method.

DIRECT_DESTRUCTIVE hard_04: User Invalidation HARD

Description: User requests a modification, then mid-execution changes their mind and wants to cancel instead. The modification's precondition becomes irrelevant.

User instruction: "Change order #W3456 to express shipping... actually wait, just cancel it instead."

Goal A (initial): modify_order_shipping(#W3456, "express") PRE: order_pending(#W3456) DEL: — ADD: shipping_updated(#W3456) Goal B (supersedes A): cancel_order(#W3456) PRE: order_pending(#W3456) DEL: order_pending(#W3456) ADD: order_cancelled(#W3456) User intent change: Goal A is INVALIDATED by Goal B.

DTG analysis: Goal B renders Goal A pointless — why modify shipping on an order that will be cancelled? This is not just an ordering problem but a goal revision problem. The agent must recognize that the user's second statement supersedes the first.

Correct resolution: Abandon Goal A entirely; execute only Goal B. The agent must track user intent changes over the conversation.

TRANSITIVE_DESTRUCTIVE hard_05: Cascading Delete VERY HARD

Description: Returning an order triggers warranty voiding, which prevents a separate warranty claim on a related product.

User instruction: "Return order #W7890 (laptop) and file a warranty claim for the charger that came with it."

Goal A: return_items(#W7890, [laptop]) PRE: order_delivered(#W7890) DEL: order_delivered(#W7890), warranty_active(laptop) ← side effect! ADD: order_returned(#W7890) Goal B: file_warranty_claim(charger, warranty=laptop_bundle) PRE: warranty_active(laptop) DEL: — ADD: warranty_claim_filed(charger) Transitive chain: return(laptop) → DELETES warranty_active(laptop) → warranty_claim(charger) NEEDS warranty_active(laptop) → FAILS

DTG analysis: This is a cross-DTG side effect. The return action on order_status also causes a transition on warranty_status (active → voided). The agent must trace the causal chain: return order → void warranty → claim fails. The dependency is transitive and not visible from the order_status DTG alone.

Correct ordering: B before A (file warranty claim first, then return). But the agent must first discover the cross-DTG dependency.

NON_SERIALIZABLE hard_06: Mutual Exclusion HARD

Description: Apply both free shipping promotion and express shipping upgrade to the same order. These are mutually exclusive shipping methods.

User instruction: "Apply my free shipping coupon FREESHIP to order #W4444 and also upgrade it to express delivery."

Goal A: apply_coupon(#W4444, FREESHIP) PRE: order_pending(#W4444), shipping_method(#W4444, standard) DEL: shipping_method(#W4444, standard) ADD: shipping_method(#W4444, free) Goal B: upgrade_shipping(#W4444, express) PRE: order_pending(#W4444), shipping_method(#W4444, standard) DEL: shipping_method(#W4444, standard) ADD: shipping_method(#W4444, express) Both need shipping_method=standard, both delete it.

DTG analysis: The shipping_method fluent has a fork from standard to both free and express. Both transitions delete the source value. A before B: shipping becomes free, B needs standard → fails. B before A: shipping becomes express, A needs standard → fails. No valid serialization.

Correct resolution: Inform the user that free shipping and express shipping are mutually exclusive. Ask which one they prefer.

DIRECT_DESTRUCTIVE hard_07: Multi-step DTG Chain MEDIUM

Description: User wants to upgrade loyalty tier from Silver to Gold to Platinum. Each step requires the previous tier as a precondition.

User instruction: "I want to upgrade my account to Platinum tier."

Goal A: upgrade_tier(user, Silver → Gold) PRE: tier(user, Silver), points(user) >= 5000 DEL: tier(user, Silver) ADD: tier(user, Gold) Goal B: upgrade_tier(user, Gold → Platinum) PRE: tier(user, Gold), points(user) >= 10000 DEL: tier(user, Gold) ADD: tier(user, Platinum) DTG chain: Silver ──[A]──→ Gold ──[B]──→ Platinum

DTG analysis: The tier fluent has a linear chain. Goal B requires the output of Goal A as its precondition. B before A: tier is Silver, B needs Gold → fails. Only A then B works. This is a causal link — A produces what B needs.

Correct ordering: A then B (strict sequential). The agent must decompose "upgrade to Platinum" into the two-step chain.

REQUIRES_OBSERVATION hard_08: Conditional Goal MEDIUM

Description: User wants to undo an order, but the action depends on the order's current status: if shipped/delivered, return it; if pending, cancel it.

User instruction: "I don't want order #W5555 anymore. Get rid of it."

IF order_status(#W5555) == pending: Goal: cancel_order(#W5555) PRE: order_pending(#W5555) ELIF order_status(#W5555) == delivered: Goal: return_items(#W5555, all) PRE: order_delivered(#W5555) ELSE (shipped): Goal: IMPOSSIBLE — cannot cancel shipped, cannot return undelivered → Must inform user to wait for delivery

DTG analysis: The correct action depends on the current node in the DTG. The agent must first observe order_status to determine which edge is available. From pending, the cancel edge exists. From delivered, the return edge exists. From shipped, no user-available edge exists — the agent is stuck and must communicate this.

Correct resolution: Observe first, then branch. This tests whether the agent can handle partial observability and status-dependent goal selection.

NON_SERIALIZABLE hard_09: Dead-End Recovery HARD

Description: Fix a device's connectivity without performing a factory reset (user wants to preserve data). The fix requires toggling airplane mode, but this destroys the current custom APN configuration.

User instruction: "My phone can't connect to the network. Fix it without factory reset. I have a custom APN I need to keep."

Goal A: fix_connectivity(device) Method: toggle airplane_mode ON then OFF (network reset) PRE: airplane_mode(off) EFFECT: forces mobile_data=off → destroys custom APN context Goal B: preserve_apn(device, custom_1) PRE: apn_config(device, custom_1), mobile_data(on) Needs mobile_data=on to verify APN is active Cross-DTG conflict: airplane_mode ON → mobile_data forced OFF → APN config inaccessible Restoring mobile_data ON → APN may reset to default

DTG analysis: Fixing connectivity requires traversing the airplane_mode DTG (off → on → off), but this causes a cross-DTG side effect that destroys mobile_data state, which in turn makes apn_config inaccessible. The agent cannot fix connectivity without risking APN loss. Neither "fix first" nor "preserve first" fully works.

Correct resolution: Record the custom APN settings before toggling airplane mode, then restore them after. This requires a save-restore pattern that goes beyond simple action ordering.

DIRECT_DESTRUCTIVE hard_10: Cross-DTG Sync Chain HARD

Description: Enable roaming with a custom APN while airplane mode is on. Requires a specific sequence across three fluent DTGs.

User instruction: "Turn off airplane mode, enable roaming, and set APN to custom_1."

Goal A: set_airplane_mode(device, off) PRE: airplane_mode(on) DEL: airplane_mode(on) ADD: airplane_mode(off) SIDE: unblocks wifi, mobile_data, roaming Goal B: enable_roaming(device) PRE: airplane_mode(off), roaming(off) DEL: roaming(off) ADD: roaming(on) Goal C: set_apn(device, custom_1) PRE: mobile_data(on), apn_config(default) DEL: apn_config(default) ADD: apn_config(custom_1) Implicit sub-goal: enable_mobile_data(device) PRE: airplane_mode(off) Needed before Goal C can execute Ordering chain: A → enable_mobile_data → C, and A → B (parallel after A)

DTG analysis: Three DTGs are connected by conditional transitions. Goal B and Goal C both depend on Goal A (airplane_mode must be off first). Goal C additionally requires mobile_data=on, which is a separate action. The agent must discover the implicit sub-goal (enable mobile data) that is not in the user's request.

Correct ordering: A first, then enable_mobile_data, then B and C can proceed in parallel. The agent must synthesize the missing intermediate step.

RESOURCE_CONSUMPTION hard_11: Counter Fluent with Lockout HARD

Description: User needs to verify identity via PIN, but has only 1 attempt remaining before account lockout. Two operations both require PIN verification.

User instruction: "I need to change my account password and also update my payment method. Both require PIN verification."

State: pin_attempts_remaining(user) = 1 Goal A: change_password(user) PRE: pin_verified(user) OR pin_attempts_remaining(user) >= 1 If PIN wrong: pin_attempts_remaining -= 1 If PIN right: pin_verified(user) = true Goal B: update_payment(user) PRE: pin_verified(user) OR pin_attempts_remaining(user) >= 1 If PIN wrong: pin_attempts_remaining -= 1 If PIN right: pin_verified(user) = true Counter fluent: pin_attempts_remaining ∈ {0, 1, 2, 3} At 0: account_locked(user) = true → BOTH goals impossible

DTG analysis: The PIN attempts counter is a strictly decreasing counter fluent. Each failed attempt is irreversible. With only 1 attempt left, the agent must verify the PIN ONCE (correctly) and then use the verified session for both operations. If the agent naively tries to verify separately for each goal, a single failure locks the account permanently.

Correct resolution: Verify PIN once, then execute both goals under the verified session. The agent must recognize the shared resource constraint and batch the verification.

MIXED hard_12: Compound Same-Entity Operations MEDIUM-HARD

Description: Three modifications plus a cancellation on the same order. The modifications are SYNERGY with each other but all are DESTRUCTIVE with the cancel.

User instruction: "For order #W9999: change the address to 456 Oak Ave, change payment to Visa ending 1234, add gift wrapping, and then cancel the whole order."

Goal A: modify_address(#W9999, "456 Oak Ave") PRE: order_pending(#W9999) DEL: — ADD: address_updated Goal B: modify_payment(#W9999, Visa-1234) PRE: order_pending(#W9999) DEL: — ADD: payment_updated Goal C: add_gift_wrap(#W9999) PRE: order_pending(#W9999) DEL: — ADD: gift_wrap_added Goal D: cancel_order(#W9999) PRE: order_pending(#W9999) DEL: order_pending(#W9999) ADD: order_cancelled Interaction matrix: A-B: SYNERGY (both self-loop at pending) A-C: SYNERGY B-C: SYNERGY A-D: DESTRUCTIVE (D deletes A's precondition) B-D: DESTRUCTIVE (D deletes B's precondition) C-D: DESTRUCTIVE (D deletes C's precondition)

DTG analysis: Goals A, B, C are all self-loops at pending (SYNERGY — any order works). Goal D is the absorbing transition to cancelled. All three modifications are DESTRUCTIVE with the cancel. But the real insight: if the user ultimately wants to cancel, why modify at all? The agent must reason about goal utility — the modifications are pointless if cancellation follows.

Correct resolution: Ask the user whether they really want to modify before cancelling (the modifications would be wasted). If yes: A, B, C (any order), then D last. Likely: just D.

Section 5: Key Findings

  1. Current benchmarks avoid hard interactions by design. τ-bench retail is 85.7% INDEPENDENT, airline is 73.8% INDEPENDENT. Tasks are constructed so that goals operate on different entities, avoiding any need for conflict reasoning. The hardest interactions (NON_SERIALIZABLE, resource consumption, cascading effects) are entirely absent.
  2. DTG analysis reveals rich structure that benchmarks don't exploit. The retail domain's order_status DTG has 3 absorbing states, a fork, and multiple self-loops — the structural ingredients for DESTRUCTIVE and NON_SERIALIZABLE interactions. Yet no task actually creates these conflicts. The structure is there; the tasks don't use it.
  3. Absorbing states create non-ergodic actions — mistakes can't be undone. Once an order is cancelled or returned, there is no edge back. This is fundamentally different from Blocksworld, where every action is reversible (stack/unstack). Absorbing states make conflict detection critical: executing the wrong action first permanently destroys the ability to achieve other goals.
  4. Cross-DTG dependencies create ordering chains that require planning. In telecom, toggling airplane mode cascades through wifi, mobile_data, roaming, hotspot, vpn, and APN configuration. An agent must understand the causal graph across fluents to find a valid execution order. This is the planning equivalent of understanding transitive dependencies in a build system.
  5. Counter fluents and resource consumption add a dimension that binary fluents miss. Gift card balances, PIN attempts, and API rate limits are monotonically decreasing counters. Unlike binary toggles (on/off), they cannot be restored. This creates resource conflicts that are invisible to boolean state analysis but critical for real-world agent behavior.
  6. The 12 hard instances cover all identified gaps. Each instance targets a specific missing interaction pattern, with difficulty calibrated from MEDIUM (single-pair ordering) to VERY HARD (transitive cross-DTG cascading effects). Together, they form a comprehensive supplement to the existing benchmarks.

Bottom line

DTG analysis is a principled method for benchmark gap analysis. Instead of guessing what scenarios are hard, we enumerate the state space topology, identify which structural features (absorbing states, forks, cross-DTG edges, counters) create hard interactions, and verify whether existing tasks exercise them. For τ-bench and τ2-bench, the answer is clear: the domain structure supports hard interactions, but the task design systematically avoids them.

DTG formalism: Helmert (2006), "The Fast Downward Planning System." Subgoal interaction taxonomy: Korf (1987), Barrett & Weld (1994). Benchmark domains: τ-bench (Yao et al. 2024), τ2-bench (Fan et al. 2025).
